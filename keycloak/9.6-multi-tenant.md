# معماری چندمستاجری (Multi‑Tenant) در سیستم ISO‑27001 Audit

## مقدمه

این سند معماری **چندمستاجری (Multi‑Tenant)** سیستم SaaS ما را توضیح می‌دهد.  
در این سیستم از **Keycloak** به عنوان ارائه‌دهنده هویت (Identity Provider) استفاده می‌شود و Backend با **FastAPI** پیاده‌سازی شده است.

اهداف این معماری:

- جداسازی کامل داده‌های هر سازمان
- جلوگیری از جعل tenant توسط frontend
- ساده نگه داشتن احراز هویت
- مستقل بودن معماری از Keycloak
- مدیریت دسترسی کاربران با RBAC

---

# مفهوم Tenant

در یک سیستم SaaS هر مشتری یک **Tenant** محسوب می‌شود.

Tenant واقعی در دیتابیس نگهداری می‌شود.

مثال:

- Aurora
- Javid
- Acme

این داده‌ها در جدول زیر ذخیره می‌شوند:

```
organizations
```

---

# تفاوت Tenant واقعی با Group در Keycloak

در Keycloak از **Group** فقط برای تشخیص عضویت کاربر در یک سازمان استفاده می‌کنیم.

مثال:

```
/org-aurora
/org-javid
/org-acme
```

اما این گروه‌ها Tenant واقعی نیستند.

تعریف درست معماری:

Tenant واقعی:
```
organizations.id
```

گروه در Keycloak:
```
membership identity
```

---

# ساختار کلی معماری سیستم

```
Keycloak
   │
   │ JWT Token
   │ (roles / groups / username)
   ▼
Backend (FastAPI)
   │
   │ resolve tenant
   ▼
Database (SQLAlchemy)
   │
   ├── organizations
   ├── users
   └── memberships
```

---

# جدول organizations

این جدول Tenant واقعی سیستم را نگهداری می‌کند.

فیلدهای پیشنهادی:

- id
- name
- slug
- keycloak_group_path
- created_at
- updated_at

مثال:

| id | name | slug | keycloak_group_path |
|----|------|------|---------------------|
| 1 | Aurora | aurora | /org-aurora |
| 2 | Javid | javid | /org-javid |

---

# مفهوم User

User یک فرد است که وارد سیستم می‌شود.

هر کاربر دارای موارد زیر است:

- username
- password
- role
- عضویت در organization

---

# Role ها در سیستم

در سیستم چند نقش اصلی تعریف شده است:

```
system
org_admin
editor
viewer
```

این نقش‌ها در client زیر در Keycloak تعریف می‌شوند:

```
iso-audit-api
```

---

# توضیح Role ها

## system

کاربر سطح سیستم است.

توانایی‌ها:

- دسترسی به همه tenant ها
- ساخت organization جدید
- ساخت user برای همه tenant ها

---

## org_admin

ادمین یک organization خاص است.

توانایی‌ها:

- مدیریت کاربران همان organization
- ساخت user جدید

محدودیت:

- نمی‌تواند system user بسازد

---

## editor

می‌تواند:

- داده ایجاد کند
- داده ویرایش کند

---

## viewer

فقط می‌تواند داده‌ها را مشاهده کند.

---

# تشخیص Tenant در Backend

Tenant از frontend دریافت نمی‌شود.

Tenant از داخل **JWT Token** استخراج می‌شود.

نمونه payload:

```json
{
 "sub": "123456",
 "preferred_username": "javid",
 "groups": [
   "/org-javid"
 ],
 "resource_access": {
   "iso-audit-api": {
     "roles": ["editor"]
   }
 }
}
```

---

# فرآیند استخراج Tenant

Backend مراحل زیر را انجام می‌دهد:

1. اعتبارسنجی JWT
2. خواندن claim گروه‌ها
3. پیدا کردن group مربوط به tenant
4. lookup در جدول organizations

مثال:

```
groups = /org-javid
```

جستجو در دیتابیس:

```
organizations.keycloak_group_path = /org-javid
```

نتیجه:

```
organization_id = tenant واقعی
```

---

# چرا tenant از frontend گرفته نمی‌شود؟

اگر frontend چیزی مثل این بفرستد:

```
X-Tenant: aurora
```

کاربر می‌تواند آن را تغییر دهد.

به همین دلیل:

```
Source of truth = JWT
```

---

# وظیفه Frontend

Frontend فقط:

1. login با Keycloak
2. دریافت access token
3. ارسال token

```
Authorization: Bearer <token>
```

Frontend نباید tenant را ارسال کند.

---

# وظیفه Backend

Backend باید:

- JWT را verify کند
- roles را استخراج کند
- groups را بخواند
- tenant واقعی را پیدا کند
- دسترسی را کنترل کند

---

# چرخه ساخت User

وقتی admin یا system user یک کاربر می‌سازد:

اطلاعات زیر مشخص می‌شود:

- username
- password
- first_name
- last_name
- role
- organization

مثال:

```
username = ali
role = editor
organization = aurora
```

---

# فرآیند ساخت user در Keycloak

مرحله ۱

ساخت user

```
POST /admin/realms/{realm}/users
```

مرحله ۲

assign role

```
POST /users/{user_id}/role-mappings/clients/{client_uuid}
```

مرحله ۳

اضافه کردن user به group

```
PUT /users/{user_id}/groups/{group_id}
```

---

# login user

بعد از login، Keycloak یک JWT صادر می‌کند.

مثال:

```json
{
 "preferred_username": "ali",
 "groups": ["/org-aurora"],
 "resource_access": {
   "iso-audit-api": {
     "roles": ["editor"]
   }
 }
}
```

Backend tenant را تشخیص می‌دهد.
