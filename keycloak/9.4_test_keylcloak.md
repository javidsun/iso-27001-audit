# ISO-27001-AUDIT

## TASK-9.4 — ساخت (Client) برای (Backend API) و تنظیم (JWT Validation)
### Backend API Client & JWT Validation — Test Guide (Manual + Edge Cases)

---

## Objective (هدف)

Verify that:

1) Backend validates JWT using **JWKS endpoint** (اعتبارسنجی با JWKS).
2) Requests **without token** return **401 Unauthorized**.
3) Requests with **valid token** return **200 OK**.
4) Token is sent only via standard header:

`Authorization: Bearer <token>`

---

## Prerequisites (پیش‌نیاز)

Start the Docker stack:

```bash
docker compose up --build
```

Assumptions (فرضیات):

- Backend base URL (inside Docker network): `http://iso-audit-backend:8003/api/v1`
- Keycloak base URL (inside Docker network): `http://keycloak:8080`
- Realm: `iso-audit`
- Token endpoint:
  `http://keycloak:8080/realms/iso-audit/protocol/openid-connect/token`
- JWKS endpoint (public keys):
  `http://keycloak:8080/realms/iso-audit/protocol/openid-connect/certs`
- Secure test endpoint:
  `GET /api/v1/secure/ping`

> Note: these commands are executed from the `iso-audit-toolbox` container, so host ports don’t matter.

## Step 4 — Obtain Access Token (Client) (Dev-only)

Your original parsing with `grep` is fragile. This version is more portable (no jq required) using `sed`.

```bash
docker exec -it iso-audit-toolbox sh -lc '
RESP=$(curl -sS -X POST \
  http://keycloak:8080/realms/iso-audit/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=iso-audit-api" \
  -d "grant_type=password" \
  -d "username=javid" \
  -d "password=Admin@@00000" \
  -d "scope=openid profile email")

TOKEN=$(printf "%s" "$RESP" | sed -n "s/.*\"access_token\":\"\\([^\"]*\\)\".*/\\1/p")

echo "TOKEN_LEN=${#TOKEN}"
test ${#TOKEN} -gt 200 && echo "TOKEN_OK" || (echo "TOKEN_MISSING" && exit 1)
'
```

Expected:
- `TOKEN_LEN > 200`
- `TOKEN_OK`

## C) JWT validation via JWKS (signature + claims)

### T9.4-C1 — Corrupted token (signature fails)
```bash
docker exec -it iso-audit-toolbox sh -lc '
RESP=$(curl -sS -X POST \
  http://keycloak:8080/realms/iso-audit/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=iso-audit-api" \
  -d "grant_type=password" \
  -d "username=javid" \
  -d "password=Admin@@00000")

TOKEN=$(printf "%s" "$RESP" | sed -n "s/.*\"access_token\":\"\\([^\"]*\\)\".*/\\1/p")
BAD="${TOKEN}X"

curl -sS -i http://iso-audit-backend:8003/api/v1/secure/ping \
  -H "Authorization: Bearer $BAD" | head -n 25
'
```
Expected: `401`

### T9.4-C2 — Token from a different realm (issuer mismatch) (if master realm accessible)
```bash
docker exec -it iso-audit-toolbox sh -lc '
RESP=$(curl -sS -X POST \
  http://keycloak:8080/realms/master/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=admin-cli" \
  -d "grant_type=password" \
  -d "username=admin_audit" \
  -d "password=admin123")

TOKEN=$(printf "%s" "$RESP" | sed -n "s/.*\"access_token\":\"\\([^\"]*\\)\".*/\\1/p")

curl -sS -i http://iso-audit-backend:8003/api/v1/secure/ping \
  -H "Authorization: Bearer $TOKEN" | head -n 25
'
```
Expected: `401`

### T9.4-C3 — Audience mismatch (if you enforce aud)
- Obtain a token for a different client (different `client_id`)
- Use it on the backend
Expected: `401`

### T9.4-C4 — Expired token
- Configure short token lifespan in Keycloak (dev) or wait until expiration
Expected: `401`

> Minimum security bar: signature + `iss` + `exp`.  
> Better: also enforce `aud` and allowed algorithms.

## D) JWKS availability / failure mode

### T9.4-D1 — JWKS endpoint down
- Stop Keycloak:
```bash
docker compose stop keycloak
```
- Call secure endpoint with any token (or even without)
Expected behavior (choose and document):
- **401** (treat as auth failure) OR
- **503** (auth infrastructure unavailable)

> Whatever you choose, be consistent. “Random behavior” is unacceptable.

### T9.4-D2 — JWKS caching (optional but recommended)
- With Keycloak up, call secure endpoint once (JWKS fetched and cached)
- Stop Keycloak
- Call secure endpoint again within cache TTL
Expected: works until cache expires (if caching implemented), otherwise fails as per D1.

# Definition of Done (DoD)

Task is COMPLETE when:

- Backend rejects requests without token (**401**)
- Backend accepts valid JWT (**200**)
- JWT signature is validated using **JWKS endpoint**
- Authorization header format is enforced: `Authorization: Bearer <token>`

# What we did in this task (چی کار کردیم و چرا)

## What we built
1) **Backend-side JWT validator** (middleware/dependency):
   - Extracts token from `Authorization: Bearer <token>`
   - Verifies JWT signature using **public keys from JWKS**
   - Validates basic claims (at least `exp` and `iss`; ideally also `aud`)
   - Rejects missing/invalid tokens with `401`

2) **A “client procedure” for dev testing**
   - A repeatable way to obtain a Keycloak token (password grant, dev-only)
   - Use that token to call a protected endpoint (`/secure/ping`)

## Why we did it this way
- **JWKS** avoids hardcoding keys in code; Keycloak can rotate keys safely.
- **Bearer standard** ensures every client uses the same convention.
- Returning **401** for missing/invalid tokens is the correct HTTP semantics.
- A dedicated `/secure/ping` endpoint lets you test auth in isolation (no business logic noise).

## Where changes usually live (typical structure)
- API layer: auth dependency/middleware (e.g., `api/auth/jwt_validator.py`)
- Adapters: JWKS fetcher client (e.g., `adapters/keycloak/jwks_client.py`)
- Config: `JWKS_URL`, `ISSUER`, optional `AUDIENCE`
- Routes: `secure/ping` protected route for verification

# ISO-27001-AUDIT

## TASK-9.4 --- توضیح فنی پیاده‌سازی JWT Validation در Backend

------------------------------------------------------------------------

## هدف این تسک چه بود؟

در این تسک هدف ما این بود که:

1.  بک‌اند بتواند توکن JWT را اعتبارسنجی کند.
2.  اعتبارسنجی از طریق JWKS Endpoint کی‌کلاک انجام شود.
3.  اگر توکن ارسال نشود → پاسخ 401 برگردانده شود.
4.  اگر توکن معتبر باشد → درخواست پذیرفته شود (200).
5.  استاندارد ارسال توکن رعایت شود:

Authorization: Bearer `<token>`{=html}

------------------------------------------------------------------------

## معماری کلی احراز هویت

### جریان کلی کار به این صورت است:

1.  کاربر در Keycloak لاگین می‌کند.
2.  Keycloak یک JWT امضا شده (RS256) تولید می‌کند.
3.  کلاینت این توکن را در هدر Authorization ارسال می‌کند.
4.  بک‌اند:
    -   هدر را بررسی می‌کند
    -   توکن را استخراج می‌کند
    -   کلید عمومی را از JWKS endpoint دریافت می‌کند
    -   امضای توکن را بررسی می‌کند
    -   issuer و exp را اعتبارسنجی می‌کند
    -   در صورت صحت → اجازه عبور می‌دهد

------------------------------------------------------------------------

## چرا از JWKS استفاده کردیم؟

اگر کلید عمومی را داخل کد هاردکد کنیم:

-   با چرخش کلید (key rotation) سیستم می‌شکند
-   امنیت کاهش پیدا می‌کند

اما با JWKS:

-   بک‌اند همیشه کلیدهای عمومی را از Keycloak می‌گیرد
-   اگر کلید عوض شود، سیستم همچنان کار می‌کند
-   این روش استاندارد OpenID Connect است

------------------------------------------------------------------------

## ساختار پیاده‌سازی در پروژه

### 1️⃣ استخراج توکن

ابتدا بررسی می‌کنیم:

-   آیا هدر Authorization وجود دارد؟
-   آیا با "Bearer" شروع می‌شود؟
-   آیا توکن خالی نیست؟

در غیر این صورت 401 برمی‌گردانیم.

------------------------------------------------------------------------

### 2️⃣ دریافت کلید عمومی از JWKS

از آدرس:

KEYCLOAK_JWKS_URL

کلیدها دریافت می‌شود و در حافظه کش می‌شوند (TTL = 300 ثانیه).

این باعث می‌شود: - برای هر درخواست call شبکه نزنیم - performance بهتر شود

------------------------------------------------------------------------

### 3️⃣ بررسی kid

هر JWT یک header دارد که شامل `kid` است.

ما: - kid را از توکن استخراج می‌کنیم - در لیست کلیدهای JWKS دنبال همان
kid می‌گردیم - اگر پیدا نشد → 401

این مرحله بسیار مهم است.

------------------------------------------------------------------------

### 4️⃣ اعتبارسنجی JWT

سپس:

-   امضا (Signature) بررسی می‌شود
-   issuer بررسی می‌شود
-   exp بررسی می‌شود
-   در صورت فعال بودن، audience هم بررسی می‌شود

اگر هرکدام نامعتبر باشد → 401

------------------------------------------------------------------------

### 5️⃣ ذخیره اطلاعات کاربر

در صورت موفقیت:

payload داخل request.state.user ذخیره می‌شود.

به این صورت route ها می‌توانند اطلاعات کاربر را بخوانند.

-   کنترل دقیق‌تر داریم
-   تست‌پذیری بهتر است

برای این تسک، Dependency منطقی‌تر و تمیزتر است.
